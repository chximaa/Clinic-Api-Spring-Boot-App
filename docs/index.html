📦 Packages Java — pourquoi cette organisation ?

Spring Boot utilise le pattern MVC en couches (Layered Architecture). Chaque package a une responsabilité unique :

  • model/ → Les données (ce qu'on stocke)
  • dao/ → Comment on stocke (accès BDD)
  • service/ → Ce qu'on fait avec les données
  • web/controller/ → Comment on expose les données à l'extérieur
  • web/exceptions/ → Comment on gère les erreurs
💡 Cette séparation s'appelle le principe de Séparation des Préoccupations (SoC). Si demain tu changes de BDD, tu ne touches qu'au DAO. Si tu changes la logique métier, tu ne touches qu'au service.

🗂️ resources/ — les fichiers de configuration

  • application.properties → Toute la config Spring (BDD, ports, Swagger…)
  • schema.sql → Crée la table SQL au démarrage
  • data.sql → Insère des données de test au démarrage
⚠️ Ces fichiers sont lus automatiquement par Spring Boot au démarrage. Tu n'as rien à appeler manuellement.

🏭 Package principal : com.clinic.clinic_api

C'est le package racine. Spring Boot scanne automatiquement tous les sous-packages pour trouver les classes annotées (@Component, @Service, etc.).

🚨 Important : La classe principale ClinicApiApplication doit être dans le package racine, sinon Spring ne trouve pas les autres classes !

🚀 La classe principale

Le point d'entrée de toute application Spring Boot.

package com.clinic.clinic_api; import org.springframework.boot.SpringApplication; import org.springframework.boot.autoconfigure.SpringBootApplication; @SpringBootApplication public class ClinicApiApplication { public static void main(String[] args) { SpringApplication.run(ClinicApiApplication.class, args); } }

🔬 @SpringBootApplication — que fait-elle exactement ?

Cette annotation est une méta-annotation qui combine 3 annotations en une seule :

@SpringBootConfiguration Marque cette classe comme configuration Spring (équivalent de @Configuration)
@EnableAutoConfiguration Active la magie Spring Boot : détecte automatiquement tes dépendances et configure tout. Ex: tu as H2 dans le classpath → Spring configure automatiquement la datasource H2.
@ComponentScan Scanne récursivement tous les packages depuis ce package pour trouver les @Component, @Service, @Repository, @Controller…

⚙️ SpringApplication.run() — que se passe-t-il ?

  1. Spring Boot lit application.properties
  2. Spring crée le contexte d'application (un gros conteneur d'objets)
  3. Spring scanne les packages et crée les beans (PatientService, PatientController…)
  4. Spring injecte les dépendances (@Autowired, injection par constructeur)
  5. Hibernate lit les @Entity et configure le mapping
  6. Spring exécute schema.sql puis data.sql
  7. Le serveur Tomcat démarre sur le port 8080
  8. ✅ L'API est prête à recevoir des requêtes !

🗄️ Entity / Model — Patient.java

La classe qui représente tes données. Le pont entre Java et SQL.

🤔 C'est quoi une Entity ?

Une Entity est une classe Java qui représente une table en base de données. Chaque instance de la classe = une ligne dans la table. Chaque champ Java = une colonne SQL.

Analogie : Si la BDD est un Excel, l'Entity est la définition des colonnes du tableau. Un objet Patient = une ligne remplie.

@Entity ← Dit à JPA : "cette classe = une table" @Table(name = "PATIENT") ← Nom exact de la table SQL @Schema(description = "Entité représentant un patient") ← Doc Swagger public class Patient { @Id ← Clé primaire @GeneratedValue(strategy = GenerationType.IDENTITY) ← AUTO_INCREMENT private Long id; @NotBlank(message = "Le nom est obligatoire") @Size(min = 2, max = 100) private String nom; @NotBlank(message = "Le prénom est obligatoire") @Size(min = 2, max = 100) private String prenom; @NotNull @Pattern(regexp = "\\d{15}", message = "Doit contenir exactement 15 chiffres") private String numeroSecu; @JsonIgnore ← Ce champ N'apparaît PAS dans le JSON renvoyé ! private String diagnosticConfidentiel; // Constructeurs, Getters, Setters, toString()... }

@JsonIgnore — la sécurité !

Le diagnostic est confidentiel. Sans @JsonIgnore, il apparaîtrait dans la réponse JSON et n'importe qui pourrait le voir via l'API. Avec @JsonIgnore, Jackson (le convertisseur JSON) ignore complètement ce champ.

✅ Le champ existe bien en BDD (les tests le prouvent), mais il est masqué dans l'API.

Validation Bean (@Valid)

Les annotations @NotBlank, @Size, @Pattern définissent des contraintes. Elles sont activées par @Valid dans le contrôleur.

  • @NotBlank → pas null ET pas vide
  • @NotNull → pas null seulement
  • @Size(min,max) → longueur du String
  • @Pattern(regexp) → doit matcher ce regex

GenerationType.IDENTITY

L'ID est généré automatiquement par la BDD (AUTO_INCREMENT). Tu n'as jamais besoin de le fournir lors de la création.

💾 DAO / Repository — PatientRepository.java

La couche qui communique directement avec la base de données.

🤔 DAO vs Repository — quelle différence ?

DAO (Data Access Object) est le pattern de conception général qui sépare la logique d'accès aux données du reste.

Repository est l'implémentation Spring/JPA de ce pattern. Dans Spring, on dit "Repository" mais c'est la même idée.

Analogie : Le DAO est le guichetier d'une banque. Tu lui demandes "donne-moi le solde du compte X" et il va chercher dans les archives (BDD) sans que toi (le service) tu saches comment.

🔮 La magie de JpaRepository

JpaRepository<Patient, Long> te donne automatiquement sans écrire une seule ligne de code :

  • findAll() → SELECT * FROM PATIENT
  • findById(id) → SELECT * WHERE id=?
  • save(p) → INSERT ou UPDATE
  • deleteById(id) → DELETE WHERE id=?
  • count() → SELECT COUNT(*)
  • existsById(id) → SELECT 1 WHERE id=?
  • Et des dizaines d'autres !
@Repository public interface PatientRepository extends JpaRepository<Patient, Long> { // 1. Dérivation de méthode — Spring génère le SQL automatiquement Optional<Patient> findByNumeroSecu(String numeroSecu); // → SELECT * FROM PATIENT WHERE NUMERO_SECU = ? // 2. Requête JPQL custom avec @Query @Query("SELECT p FROM Patient p WHERE LOWER(p.nom) LIKE LOWER(CONCAT('%', :nom, '%'))") List<Patient> rechercherParNom(@Param("nom") String nom); // → Recherche insensible à la casse avec wildcard // 3. JPQL avec deux paramètres @Query("SELECT p FROM Patient p WHERE LOWER(p.nom)=LOWER(:nom) AND LOWER(p.prenom)=LOWER(:prenom)") Optional<Patient> findByNomAndPrenom(@Param("nom") String nom, @Param("prenom") String prenom); }

Dérivation de noms (Query Methods)

Spring lit le nom de la méthode et génère le SQL. Les mots-clés reconnus :

  • findBy → SELECT WHERE
  • And → AND
  • Or → OR
  • Like → LIKE
  • OrderBy → ORDER BY
  • Not → NOT
Ex: findByNomAndPrenom → SELECT WHERE NOM=? AND PRENOM=?

JPQL vs SQL natif

JPQL utilise les noms des classes Java (pas des tables SQL) :

SELECT p FROM Patient p WHERE p.nom…

SQL natif utilise les noms SQL :

SELECT * FROM PATIENT WHERE NOM…
Le projet utilise JPQL → plus portable, indépendant du SGBD.

Optional<T> — pourquoi ?

Optional signifie que le résultat peut être absent. Au lieu de retourner null (dangereux → NullPointerException), on retourne un Optional vide.

Dans le service : .orElseThrow() lance une exception propre si vide.

Hiérarchie de JpaRepository

JpaRepository<T, ID>
↑
PagingAndSortingRepository
↑
CrudRepository
↑
Repository (marker interface)

Chaque niveau ajoute des méthodes. JpaRepository est le plus complet.

⚡ Service — PatientService.java

La couche métier : le cerveau de l'application.

🤔 Pourquoi une couche Service ?

Le Service contient la logique métier = les règles spécifiques à ton domaine. Il fait le lien entre le Controller (qui reçoit les requêtes HTTP) et le DAO (qui parle à la BDD).

Analogie : Le Controller est le serveur d'un restaurant (prend la commande), le Service est le cuisinier (prépare le plat avec les règles de la cuisine), le DAO est le garde-manger (fournit les ingrédients).

@Service public class PatientService { // Injection par constructeur (BONNE PRATIQUE vs @Autowired sur le champ) private final PatientRepository repository; public PatientService(PatientRepository repository) { this.repository = repository; ← Spring injecte automatiquement } // ===== READ ===== public List<Patient> getAllPatients() { return repository.findAll(); ← JpaRepository → SELECT * FROM PATIENT } public Patient getPatientById(Long id) { return repository.findById(id) .orElseThrow(() -> new PatientNotFoundException(id)); ← Si absent → exception 404 (gérée par GlobalExceptionHandler) } // ===== CREATE ===== public Patient createPatient(Patient patient) { return repository.save(patient); ← INSERT INTO PATIENT... } // ===== UPDATE ===== public Patient updatePatient(Long id, Patient updated) { Patient existing = getPatientById(id); ← Vérifie existence (404 si absent) existing.setNom(updated.getNom()); existing.setPrenom(updated.getPrenom()); existing.setNumeroSecu(updated.getNumeroSecu()); existing.setDiagnosticConfidentiel(updated.getDiagnosticConfidentiel()); return repository.save(existing); ← UPDATE PATIENT SET... WHERE id=? } // ===== DELETE ===== public void deletePatient(Long id) { getPatientById(id); ← Vérifie d'abord que le patient existe ! repository.deleteById(id); ← DELETE FROM PATIENT WHERE id=? } // ===== RECHERCHE MÉTIER ===== public Patient findByNumeroSecu(String secu) { return repository.findByNumeroSecu(secu) .orElseThrow(() -> new PatientNotFoundException("secu = " + secu)); } public List<Patient> findByNom(String nom) { return repository.rechercherParNom(nom); } }

🔑 Point crucial : Injection par constructeur vs @Autowired sur champ

❌ Mauvaise pratique (champ)

@Autowired private PatientRepository repo;
  • Difficile à tester (on ne peut pas injecter facilement un mock)
  • Dépendances cachées
  • Le champ peut être null au démarrage

✅ Bonne pratique (constructeur)

private final PatientRepository repo; PatientService(PatientRepository repo) { this.repo = repo; }
  • Testable facilement (inject un mock dans le constructeur)
  • Dépendances explicites
  • final → immutable, thread-safe

🎮 Controller — PatientController.java

La façade de ton API. Il reçoit les requêtes HTTP du monde extérieur.

🤔 Qu'est-ce qu'un Controller REST ?

Le Controller est la porte d'entrée de ton application. Il écoute les requêtes HTTP, lit les paramètres, délègue au Service, et renvoie une réponse JSON.

Analogie : Le réceptionniste d'un hôtel. Il ne fait pas le ménage (Service) ni ne gère les clés lui-même (DAO). Il accueille, oriente, et répond.

@RestController ← @Controller + @ResponseBody @RequestMapping("/api/patients") ← Préfixe commun à tous les endpoints @Tag(name = "Patients", description = "API de gestion des patients") public class PatientController { private final PatientService service; public PatientController(PatientService service) { this.service = service; } // GET /api/patients → liste tous @GetMapping public ResponseEntity<List<Patient>> getAll() { return ResponseEntity.ok(service.getAllPatients()); } // GET /api/patients/5 → un patient par ID @GetMapping("/{id}") public ResponseEntity<Patient> getById(@PathVariable Long id) { return ResponseEntity.ok(service.getPatientById(id)); } // POST /api/patients → crée un patient @PostMapping public ResponseEntity<Patient> create(@Valid @RequestBody Patient patient) { Patient created = service.createPatient(patient); return ResponseEntity.status(HttpStatus.CREATED).body(created); ← Status 201 CREATED (pas 200 !) pour les créations } // PUT /api/patients/5 → modifie un patient @PutMapping("/{id}") public ResponseEntity<Patient> update(@PathVariable Long id, @Valid @RequestBody Patient patient) { return ResponseEntity.ok(service.updatePatient(id, patient)); } // DELETE /api/patients/5 → supprime @DeleteMapping("/{id}") public ResponseEntity<Void> delete(@PathVariable Long id) { service.deletePatient(id); return ResponseEntity.noContent().build(); ← Status 204 NO CONTENT (rien à retourner) } // GET /api/patients/search/secu?secu=175126012345699 @GetMapping("/search/secu") public ResponseEntity<Patient> searchBySecu(@RequestParam String secu) { return ResponseEntity.ok(service.findByNumeroSecu(secu)); } // GET /api/patients/search/nom?nom=Dup @GetMapping("/search/nom") public ResponseEntity<List<Patient>> searchByNom(@RequestParam String nom) { return ResponseEntity.ok(service.findByNom(nom)); } }

// 1. Exception métier personnalisée public class PatientNotFoundException extends RuntimeException { public PatientNotFoundException(Long id) { super("Patient non trouvé avec l'id : " + id); } public PatientNotFoundException(String message) { super("Patient non trouvé : " + message); } } // 2. Handler global qui intercepte TOUTES les exceptions @RestControllerAdvice ← Intercepte les exceptions de TOUS les controllers public class GlobalExceptionHandler { @ExceptionHandler(PatientNotFoundException.class) public ResponseEntity<ErrorResponse> handleNotFound(PatientNotFoundException ex) { return ResponseEntity .status(HttpStatus.NOT_FOUND) .body(new ErrorResponse(404, ex.getMessage())); } @ExceptionHandler(MethodArgumentNotValidException.class) ← validation @Valid public ResponseEntity<ErrorResponse> handleValidation(MethodArgumentNotValidException ex) { String errors = ex.getBindingResult().getFieldErrors().stream() .map(e -> e.getField() + ": " + e.getDefaultMessage()) .collect(Collectors.joining(" | ")); return ResponseEntity.status(HttpStatus.BAD_REQUEST).body(new ErrorResponse(400, errors)); } @ExceptionHandler(Exception.class) ← Catch-all pour le reste public ResponseEntity<ErrorResponse> handleGeneral(Exception ex) { return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR) .body(new ErrorResponse(500, "Erreur interne: " + ex.getMessage())); } } // 3. Structure de la réponse d'erreur public class ErrorResponse { private LocalDateTime timestamp; private int status; private String message; public ErrorResponse(int status, String message) { this.timestamp = LocalDateTime.now(); this.status = status; this.message = message; } // Getters... }

💡 @RestControllerAdvice vs @ControllerAdvice ?

@RestControllerAdvice = @ControllerAdvice + @ResponseBody

Les méthodes retournent directement du JSON (pas de vue HTML).

Exemple de réponse d'erreur 404 :

{ "timestamp": "2024-02-27T15:30:00", "status": 404, "message": "Patient non trouvé avec l'id : 999" }

🛢️ Base de données — H2, schema.sql, data.sql

Comment la persistance des données est configurée.

schema.sql — Création de la table

-- Supprime si elle existe (évite les erreurs au redémarrage) DROP TABLE IF EXISTS PATIENT; CREATE TABLE PATIENT ( id BIGINT AUTO_INCREMENT PRIMARY KEY, nom VARCHAR(100) NOT NULL, prenom VARCHAR(100) NOT NULL, numero_secu VARCHAR(15) NOT NULL UNIQUE, diagnostic_confidentiel VARCHAR(500) );
⚠️ ddl-auto=none → Hibernate ne touche pas au schéma. Spring Boot exécute ce fichier SQL à chaque démarrage.

🔬 H2 en mémoire — Avantages et limites

✅ Avantages

  • Aucune installation nécessaire
  • Ultra rapide (mémoire RAM)
  • Parfait pour les tests et le dev
  • Réinitialisation automatique au redémarrage
  • Console web intégrée

❌ Limites

  • Données perdues à l'arrêt
  • Ne convient pas à la production
  • Pour prod : MySQL, PostgreSQL, Oracle…

@SpringBootTest

Les méthodes d'assertion (JUnit 5)

  • assertEquals(expected, actual) → valeurs égales
  • assertNotNull(obj) → pas null
  • assertThrows(Exception.class, lambda) → exception lancée
  • assertTrue(condition) → condition vraie
